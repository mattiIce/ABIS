# ABIS — Data Model (partial, recovered)

> **Status:** partial model, machine-extracted. **Source:** the database columns,
> types, and joins embedded in the **exported DataWindow/query sources**
> (`*.srd`, `*.srq`) committed to this repo. **Not** a live database
> introspection. Regenerate with `python3 tools/extract_schema.py .`
> (writes [`docs/data-model/schema.json`](data-model/schema.json)).

## Why this is partial

Almost all ABIS logic lives in compiled, binary `.pbl` libraries that cannot be
parsed here. Only ~40 DataWindows and a couple of SQL queries exist as readable
text exports, so this model only covers the **20 tables / 140 columns** those
objects happen to touch. The real database is much larger (the inventory shows
~1,900 DataWindows app-wide — see [`OBJECT_INVENTORY.md`](OBJECT_INVENTORY.md)).
Treat this as a verified seed for schema reconstruction, not the whole schema.

Types are PowerBuilder DataWindow column types (`decimal(n)` is a DW display
spec, not necessarily the DB precision; `char(n)` widths come from the DW).

## Domain shape

The recovered tables center on a metal coil → sheet **processing/slitting**
operation:

- **`ab_job`** — the central *production job* entity (a job runs on a `line_num`,
  belongs to an order via `order_abc_num`, has start/finish timestamps, a
  `material_yield`, and status). Everything on the shop floor hangs off a job.
- **`coil`** — raw input coils (alloy, gauge, width, weights, location, owning
  `customer_id`). Identified by `coil_abc_num`.
- **`process_coil`** — links a `coil` to the `ab_job` consuming it (status,
  processed weight/quantity). The job ↔ coil junction.
- **`sheet_skid`** / **`process_partial_skid`** — finished/partial output skids
  produced by a job (`ab_job_num`), with weights and piece counts.
- **`scrap_skid`** / **`return_scrap_item`** — scrap and returns by job.
- **`customer_order`** / **`order_item`** — commercial order and its line items
  (part number, alloy/temper/gauge tolerances, price, theoretical weight).
- **`customer`** / **`customer_duns_xref`** — customers (incl. DUNS for EDI).
- **`pst_test_result`** / **`temp_test_result`** — mechanical/QA test results
  (yield/tensile/elongation, `n`/`r` values, by `position`); near-identical
  columns suggest a "permanent" vs "temporary/staging" pair.
- **`reject_coil_packing_item`**, **`inbound_shipment`**, **`shipment`** —
  rejects, inbound/outbound shipping (EDI file ids, BOL).
- **`die`** — tooling. **`abis_ini`** — DB-resident application settings
  (section/key/value). **`security_application`** — app-level access control.
  **`opc_action_log`** — audit log of OPC (PLC/SCADA) actions.

## Relationships

### Explicit (from graphical-query joins)

These were read directly from `PBSELECT` join clauses:

| from | to |
|---|---|
| `production_sheet_item.prod_item_num` | `sheet_skid_detail.prod_item_num` |
| `sheet_skid_detail.sheet_skid_num` | `sheet_skid_dimension_check.sheet_skid_num` |

(`production_sheet_item`, `sheet_skid_detail`, `sheet_skid_dimension_check` come
from a query's join graph; they are not in the column-extracted table list above
because their columns weren't individually bound in a parsed DataWindow.)

### Inferred (heuristic — shared column names)

A column name shared across tables is a strong FK hint in a legacy schema.
**Unverified** — confirm against the real DB before relying on these:

| shared column | tables | likely meaning |
|---|---|---|
| `ab_job_num` | `ab_job`, `process_coil`, `process_partial_skid`, `sheet_skid` | job is the hub; processing/output rows reference it |
| `coil_abc_num` | `coil`, `process_coil`, `reject_coil_packing_item` | coil referenced by processing & rejects |
| `order_abc_num` | `ab_job`, `customer_order` | job fulfills an order |
| `customer_id` | `coil`, `customer` | coil owned by / sourced from a customer |
| `customer_name` | `customer`, `customer_duns_xref` | DUNS cross-reference |
| `sheet_skid_num` | `sheet_skid`, `process_partial_skid` | partial skid references its sheet skid |
| `created_date`, `position`, `test_type`, `thickness`, `width` | `pst_test_result`, `temp_test_result` | parallel test tables (coincidental column names, not FKs) |

> **API-introduced inference:** the Phase-2 API models an `order_item.order_abc_num`
> FK to `customer_order` (not present in the extracted columns) because order
> entry fundamentally needs an order↔line-item link, and `ab_job` carries both
> `order_abc_num` and `order_item_num`. This is a **hypothesis to confirm** against
> the real schema — see [`../api/`](../api/README.md).
